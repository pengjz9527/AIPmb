为什么需要 AI 原生银行？

传统银行 App 的痛点

  • 功能堆砌，入口层级深，用户迷失在菜单中
  • 每个功能独立页面，上下文无法贯通
  • 搜索框只能匹配关键词，无法理解意图
  • 推荐系统基于规则，千人一面
  • 新增功能 = 新增页面 + 新增入口，边际成本递增

AI 原生银行的优势

  • 自然语言统一入口：一句话直达任意服务
  • 上下文贯通：对话即会话，历史即记忆
  • 意图理解：LLM 深度解析用户需求
  • 千人千面：基于记忆的个人化服务
  • 功能即 Skill：新增能力只需注册 Skill，零前端改动
"AI 原生银行不是'App 里加个聊天框'，而是以 LLM 为核心重新设计服务架构，让 AI 成为银行服务的操作系统。"

LLM as Controller：统一智能入口

设计思路

  • 单一智能体入口："小易"作为唯一入口，避免多 Agent 协作的复杂度
  • LLM 自主决策：通过 Function Calling 决定调用哪个 Skill/Tool
  • 评分路由兜底：AgentRegistry 评分阈值 0.3，低于阈值自动路由到小易
  • 预留扩展：支持 @智能体名 显式指定（未来可扩展多 Agent）
为什么不用多 Agent 协作？
银行场景下用户意图相对明确（查账/理财/消费等），多 Agent 路由增加延迟和错误率。单一 Agent + Skill 编排更简洁高效。

Agent 注册与路由

# pmb/agents/registry.py
class AgentRegistry:
    ROUTE_THRESHOLD = 0.3

    def route(self, user_message: str) -> BaseAgent:
        # 1. 检查显式指定（@智能体名）
        if user_message.startswith("@"):
            ...
        # 2. 各 Agent 评分，取最高分
        best_agent = None
        best_score = 0.0
        for agent in self._agents.values():
            score = agent.can_handle(user_message)
            ...
        # 3. 低于阈值路由到通用助手
        if best_score < self.ROUTE_THRESHOLD:
            return self._agents.get("general_assistant")
        return best_agent

代码示例：通用助手的流式分析

# pmb/agents/general_assistant.py
async def analyze_stream(self, context, event_queue):
    conv = context_manager.get_or_create(context.session_id)
    messages = conv.get_messages()
    messages.append({"role": "user", "content": context.user_message})

    llm = QwenLLM()
    all_tools = list(ALL_TOOLS) + skill_orchestrator.to_openai_tools()

    max_rounds = 5
    while max_rounds > 0:
        # === 阶段1: 意图识别 ===
        await event_queue.put({
            "type": "thinking_step",
            "step_id": "phase_intent",
            "display_name": "意图识别",
            "status": "invoking",
            "phase": "intent",
        })

        stream = llm.chat_stream(messages=messages, tools=all_tools)
        content_buffer = ""
        tool_calls_buffer = []

        async for chunk in stream:
            if chunk.reasoning_content:
                await event_queue.put({
                    "type": "reasoning_chunk",
                    "content": chunk.reasoning_content,
                })
            if chunk.content:
                content_buffer += chunk.content
                await event_queue.put({
                    "type": "ai_chunk",
                    "content": chunk.content,
                })
            if chunk.tool_calls:
                tool_calls_buffer.extend(chunk.tool_calls)

        # 处理 tool calls...
        if tool_calls_buffer:
            for tc in tool_calls_buffer:
                result, summary = await skill_orchestrator.execute_skill(
                    tc.name, tc.arguments, user_name=context.user_name
                )
                # 发送完成事件...
            max_rounds -= 1
            continue

        # 无 tool calls，最终回复
        await event_queue.put({
            "type": "ai_done",
            "content": content_buffer,
            "is_final": True,
        })
        break

架构设计原则

1️⃣

单一智能体入口

"小易"作为唯一入口，避免多 Agent 协作的复杂度和延迟。复杂任务通过 Skill 编排而非多 Agent 路由完成。

2️⃣

Skill 无状态纯函数

每个 Skill 独立、无状态、可测试。输入用户标识 + 参数，输出结构化数据。像微服务一样注册和编排。

3️⃣

事件驱动流式响应

通过 asyncio.Queue 事件队列实现真流式。thinking_step / reasoning_chunk / ai_chunk / ai_done 四种事件类型。

4️⃣

LLM 只做决策不做数据

LLM 负责意图识别和结果生成，所有数据操作由 Skill/Tool 完成。避免 LLM 幻觉影响数据准确性。

记忆是 AI 从"工具"升级为"伙伴"的关键。没有记忆的 AI 每次对话都是陌生人，有记忆的 AI 才能提供真正的个性化服务。

会话级记忆：20 轮上下文窗口

ContextManager 实现

  • 按 session_id 管理：每个 WebSocket 会话独立上下文
  • 自动 Trim：保留最近 20 轮对话，超出自动移除最早消息
  • System Prompt 动态注入：每次请求前注入用户记忆摘要
  • 内存存储：会话期间全内存操作，零 I/O 延迟
设计考量：20 轮是成本与效果的平衡点。超过 20 轮后 LLM 注意力分散，且 Token 成本急剧上升。 
# pmb/llm/context_manager.py
class ConversationContext:
    MAX_MESSAGES = 20

    def __init__(self):
        self.messages: list[dict] = []
        self.system_prompt: str = ""

    def add_message(self, role: str, content: str):
        self.messages.append({
            "role": role, "content": content
        })
        # 自动 Trim 保留最近 N 轮
        if len(self.messages) > self.MAX_MESSAGES:
            self.messages = self.messages[-self.MAX_MESSAGES:]

    def get_messages(self) -> list[dict]:
        result = []
        if self.system_prompt:
            result.append({
                "role": "system",
                "content": self.system_prompt
            })
        result.extend(self.messages)
        return result

压缩级记忆：LLM 智能摘要

自动压缩策略

  • 触发条件：会话结束且超过 1 天未压缩
  • 压缩方式：调用 LLM 将完整对话压缩为 ≤150 字摘要
  • 清理策略：超过 30 天的旧记忆自动清理
  • 降级机制：LLM 不可用时保留原始消息，不阻塞服务

# pmb/core/memory_service.py
async def compress_if_needed(self, user_name: str):
    """后台压缩超过1天的记忆"""
    memory = self._memories.get(user_name)
    if not memory:
        return

    for entry in memory.entries:
        if entry.compressed:
            continue
        age = datetime.now() - entry.ended_at
        if age > timedelta(days=1):
            summary = await self._generate_summary(
                entry.raw_messages
            )
            entry.summary = summary
            entry.compressed = True

async def _generate_summary(self, messages: list) -> str:
    """调用LLM生成≤150字摘要"""
    prompt = f"请总结以下对话的核心信息，"
             f"≤150字：{messages}"
    response = await llm.chat(messages=[
        {"role": "user", "content": prompt}
    ])
    return response.content[:150]

流式响应的 State 管理

Riverpod 架构

  • chatMessagesProvider：StateNotifier 管理消息列表状态
  • chatStatusProvider：枚举状态（idle/loading/streaming/error）

关键设计：WebSocketService 是全局单例，但 StreamController 懒创建且永不关闭。页面切换时 Notifier 重新监听，避免流已关闭的问题。 
// aipmb_app/lib/providers/chat_provider.dart
class ChatMessagesNotifier
    extends StateNotifier<List<ChatMessage>> {
  final WebSocketService _ws;
  StreamSubscription? _sub;

  ChatMessagesNotifier(this._ws) : super([]) {
    _sub = _ws.messageStream.listen(_onEvent);
  }

  void _onEvent(ChatEvent event) {
    switch (event.type) {
      case 'thinking_step':
        _updateThinkingStep(event);
        break;
      case 'ai_chunk':
        _appendToLastMessage(event.content);
        break;
      case 'ai_done':
        _finalizeMessage(event);
        break;
      case 'error':
        _showError(event.content);
        break;
    }
  }

  void sendMessage(String text) {
    state = [...state, ChatMessage.user(text)];
    _ws.send({"type": "chat", "message": text});
  }
}

            "content": [

                 }}
            ]
        }]
        response = await self.chat(messages)
