# Implementation Plan: AI-Powered User Tag Bar Optimization for "此刻" Page

**Date**: 2026-06-09  
**Status**: Draft  
**Target**: Replace rule-based tags with AI-generated tags on the Moment page, add tap-to-detail interaction

---

## 1. Context and Problem Statement

### Current State

The "此刻" (Moment) page displays user profile tags as plain `Chip` widgets in a `Wrap` layout. These tags are generated by `TagEngine.compute_tags()` — a pure rule-based engine that applies hardcoded thresholds (e.g., balance > 500k = "高净值", spending avg > 8000 = "高消费"). The tags are simple strings like "基础", "餐饮达人", "高消费" with no explanatory detail and no tap interaction.

Meanwhile, the project already has a fully functional **AI tagging system** (`pmb/ai_manage/services/tagging_service.py`) that:
- Generates creative, data-backed tags via LLM (QwenLLM)
- Returns tags as `[{"name": "...", "reasoning": "..."}]` with rich descriptions
- Caches results to `user_tags.json` on disk
- Is consumed by the **admin manage app** (`aipmb_manage`) via `/api/v1/manage/tags/*` endpoints

These two tagging systems are **completely disconnected** — the user-facing app never accesses the AI-generated tags.

### Goal

Unify these systems so the "此刻" page:
1. Displays **AI-generated tags** (with descriptive detail) by default
2. **Falls back** to rule-based tags if AI generation is unavailable
3. Shows a **bottom sheet with tag detail** when a tag is tapped

---

## 2. Architecture Decision: Integration Point

### Decision: Modify the API endpoint, not the TagEngine

**We will modify `GET /api/v1/profile/tags` in `recommendation_router.py`** to bridge the two systems. The TagEngine remains untouched as the fallback. No new file is needed on the backend.

**Rationale**:
- `TagEngine` is a pure rule engine — keep its contract stable
- `tagging_service` already handles LLM calls, cache, and error handling
- The router is the natural integration point that decides priority
- This avoids duplicating LLM-calling logic

### Decision: Use existing `tagging_service` cache (not TagEngine's in-memory cache)

**The AI tags use `user_tags.json` on disk** (via `pmb/ai_manage/store.py`). This is persistent across restarts, unlike TagEngine's in-memory dict. We keep both caches independent:
- `tagging_service` cache (`user_tags.json`) — for AI tags
- `TagEngine._cache` — for rule-based tags (fast in-memory fallback)

## 4. Detailed Changes Per File

### 4.1 Backend: `pmb/api/schemas/recommendation.py`

**What changes**: Add a new `AITagItem` model and extend `UserProfileTags` with an optional `ai_tags` field.

**Key structure**:

```python
class AITagItem(BaseModel):
    name: str          # e.g., "高铁通勤VIP"
    description: str   # e.g., "近3个月铁路出行消费占比38%，月均12次..."

class UserProfileTags(BaseModel):
    asset_level: str
    consumption_preference: str
    consumption_level: str
    risk_preference: str
    channel_preference: str
    top_categories: list[str]
    tags: list[str]
    # NEW FIELD - optional for backward compatibility
    ai_tags: list[AITagItem] = []
```

**Backward compatibility**: Clients not aware of `ai_tags` simply ignore it. The existing `tags` field continues to hold the flat rule-based string list for fallback rendering.

### 4.2 Backend: `pmb/api/routers/recommendation_router.py`

**What changes**: Modify the `get_profile_tags` endpoint (lines 158-164) to:
1. First try to get AI tags from `tagging_service.get_tags_for_user(user_name)`
2. If AI tags exist, merge them into the response
3. If AI tags don't exist, try to generate them (fire-and-forget or synchronous, see below)
4. Always compute rule-based tags as fallback base
5. Wrap everything in try/except for graceful degradation

**Key structure**:

```python
@router.get("/profile/tags", summary="用户画像标签")
async def get_profile_tags(user_name: str = Header("", alias="x-user-name")):
    user_name = _decode_user_name(user_name)

    # 1. Always compute rule-based tags (fast, no LLM dependency)
    from pmb.profile.tag_engine import tag_engine
    tags = tag_engine.compute_tags(user_name=user_name)

    # 2. Try to get AI tags from cache
    ai_tags = []
    try:
        from pmb.ai_manage.services.tagging_service import get_tags_for_user
        cached = get_tags_for_user(user_name)
        if cached is not None and cached.tags:
            ai_tags = [
                {"name": t["name"], "description": t.get("reasoning", t.get("description", ""))}
                for t in cached.tags
            ]
    except Exception:
        pass  # Graceful degradation: no AI tags

    # 3. If no AI tags cached, trigger async generation (non-blocking)
    #    The next request will pick up the cached result.
    if not ai_tags and user_name:
        try:
            from pmb.ai_manage.services.tagging_service import generate_tags_for_user
            import asyncio
            asyncio.create_task(generate_tags_for_user(user_name))
        except Exception:
            pass

    return ApiResponse(data={**tags, "ai_tags": ai_tags})
```

**Graceful degradation flow**:
- LLM call fails → `ai_tags` stays empty → frontend falls back to `tags` array
- `tagging_service` import fails → caught by outer try/except
- User has no consumption data → rule-based tags still provide minimal labels

### 4.4 Frontend: `aipmb_app/lib/models/user_profile.dart`

**What changes**: Add `AITagItem` class and modify `UserProfileTags` to parse the new `ai_tags` field.

**Key structure**:

```dart
class AITagItem {
  final String name;
  final String description;

  const AITagItem({required this.name, required this.description});

  factory AITagItem.fromJson(Map<String, dynamic> json) => AITagItem(
    name: json['name'] ?? '',
    description: json['description'] ?? '',
  );
}

class UserProfileTags {
  final String assetLevel;
  final String consumptionPreference;
  final String consumptionLevel;
  final String riskPreference;
  final String channelPreference;
  final List<AITagItem> aiTags;  // NEW

  const UserProfileTags({
    required this.assetLevel,
    required this.consumptionPreference,
    required this.consumptionLevel,
    required this.riskPreference,
    required this.channelPreference,
    this.aiTags = const [],
  });

  factory UserProfileTags.fromJson(Map<String, dynamic> json) => UserProfileTags(
    assetLevel: json['asset_level'] ?? '',
    consumptionPreference: json['consumption_preference'] ?? '',
    consumptionLevel: json['consumption_level'] ?? '',
    riskPreference: json['risk_preference'] ?? '',
    channelPreference: json['channel_preference'] ?? '',
    aiTags: (json['ai_tags'] as List<dynamic>?)
        ?.map((t) => AITagItem.fromJson(t as Map<String, dynamic>))
        .toList() ?? [],
  );

  // Keep for backward compatibility
  List<String> toList() => [
    assetLevel,
    consumptionPreference,
    consumptionLevel,
    riskPreference,
    channelPreference,
  ].where((t) => t.isNotEmpty).toList();

  // NEW: whether AI tags are available
  bool get hasAiTags => aiTags.isNotEmpty;

  // NEW: get display labels (AI tags preferred, fallback to rule-based)
  List<String> get displayLabels =>
      hasAiTags ? aiTags.map((t) => t.name).toList() : toList();
}
```

### 4.5 Frontend: `aipmb_app/lib/pages/moment/moment_page.dart`

**What changes**: Replace `_buildProfileTags(List<String> tags)` with `_buildProfileTags(UserProfileTags tags)` that:
1. Uses `tags.displayLabels` for the Chip labels
2. Adds `onPressed` to each Chip if AI tags are available
3. Shows `showModalBottomSheet` with tag name + description on tap

**Key structure**:

```dart
Widget _buildProfileTags(UserProfileTags tags) {
  final labels = tags.displayLabels;
  if (labels.isEmpty) return const SizedBox.shrink();

  return Wrap(
    spacing: 8,
    runSpacing: 8,
    children: List.generate(labels.length, (index) {
      final label = labels[index];
      final hasDetail = tags.hasAiTags && index < tags.aiTags.length;
      return Chip(
        label: Text(label, style: const TextStyle(fontSize: 12)),
        materialTapTargetSize: MaterialTapTargetSize.shrinkWrap,
        visualDensity: VisualDensity.compact,
        onPressed: hasDetail
            ? () => _showTagDetail(context, tags.aiTags[index])
            : null,
      );
    }),
  );
}

void _showTagDetail(BuildContext context, AITagItem tag) {
  showModalBottomSheet(
    context: context,
    shape: const RoundedRectangleBorder(
      borderRadius: BorderRadius.vertical(top: Radius.circular(16)),
    ),
    builder: (ctx) => Padding(
      padding: const EdgeInsets.all(24),
      child: Column(
        mainAxisSize: MainAxisSize.min,
        crossAxisAlignment: CrossAxisAlignment.start,
        children: [
          Center(
            child: Container(
              width: 40, height: 4,
              decoration: BoxDecoration(
                color: Colors.grey[300],
                borderRadius: BorderRadius.circular(2),
              ),
            ),
          ),
          const SizedBox(height: 20),
          Text(tag.name,
            style: const TextStyle(fontSize: 20, fontWeight: FontWeight.bold)),
          const SizedBox(height: 12),
          Text(tag.description,
            style: TextStyle(fontSize: 15, color: Colors.grey[700], height: 1.5)),
          const SizedBox(height: 24),
        ],
      ),
    ),
  );
}
```

## 6. Cache Strategy

| Layer | Mechanism | TTL | Scope |
|-------|-----------|-----|-------|
| TagEngine (rule) | In-memory dict `self._cache` | Until `clear_cache()` called (on login) | Per process |
| tagging_service (AI) | JSON file `user_tags.json` via store.py | Until `force=True` regeneration | Persistent |
| API response | No additional cache | N/A | N/A |
| Flutter Riverpod | `FutureProvider` auto-caches | Until `ref.invalidate()` (on pull-to-refresh) | Per session |

**Cache invalidation flow**:
1. User pulls to refresh → `ref.invalidate(profileTagsProvider)` → new API call
2. API call → `tagging_service.get_tags_for_user()` → checks JSON file
3. If cache hit → return immediately
4. If cache miss → trigger async generation, return rule-based tags for now; next refresh will pick up AI tags
5. On login → `tag_engine.clear_cache()` already called by `auth_router.py`; AI cache persists (it's per-user)

---

## 7. Error Handling and Graceful Degradation

| Scenario | Behavior |
|----------|----------|
| tagging_service unavailable / import error | ai_tags = [], frontend shows rule-based tags |
| LLM call fails (timeout, bad response) | Caught in tagging_service, fallback tag stored, next call serves from cache |
| JSON parse error from LLM | Fallback: `[{"name": "消费达人", "description": "基于消费数据分析"}]` |
| User has no consumption data | Rule-based tags still return "基础" etc.; LLM may produce generic tag |
| Network error on frontend | `tagsAsync.error` case in Riverpod → `SizedBox.shrink()`, no crash |

## 9. Risks and Caveats

1. **Async generation timing**: First request for a new user will return rule-based tags only. The AI tags appear on the *next* request after generation completes. This is acceptable for MVP — users will see AI tags after a pull-to-refresh or on next app open.

2. **LLM cost**: Every new user triggers one LLM call. The `user_tags.json` cache prevents repeated calls. For production with many users, consider batch pre-generation in the admin panel.

3. **Old Flutter clients**: The `ai_tags` field is additive. Old clients that don't have the updated model will simply ignore it and continue showing rule-based tags — fully backward compatible.

4. **`tagging_service` uses `ai_manage` LLM config**: The service resolves LLM via `model_config_service.get_active_config()`, which may differ from the main app's `LLM_MODEL` env var. Ensure the admin-configured model is appropriate for end-user-facing tag generation.

5. **`top_categories` field**: Currently unused in the frontend model but present in API response. No changes needed — it remains available for future use.
