# 个人金融信息保护规定

## 法规概述

| 项目 | 内容 |
|------|------|
| 法规全称 | 《中国人民银行金融消费者权益保护实施办法》及《个人金融信息保护技术规范》（JR/T 0171-2020） |
| 发布机构 | 中国人民银行 |
| 发布时间 | 实施办法2020年9月1日（中国人民银行令〔2020〕第5号）；技术规范2020年9月 |
| 上位法 | 《中华人民共和国个人信息保护法》（2021年11月1日施行） |
| 效力层级 | 部门规章 + 金融行业标准 |

个人金融信息保护是金融消费者权益保护的重要内容。我国个人金融信息保护的规范体系包括《个人信息保护法》（上位法）、中国人民银行《金融消费者权益保护实施办法》（部门规章）、《个人金融信息保护技术规范》（行业标准）以及《数据安全法》等多个法律法规。其中，C1-C5分类保护体系是个人金融信息保护的核心框架。

## 核心内容摘要

### 一、个人金融信息的定义

个人金融信息是指金融机构通过开展业务或者其他合法渠道获取、存储和处理的消费者个人金融信息，包括但不限于：
- 个人身份信息（姓名、身份证号、联系方式等）
- 个人财产信息（存款、贷款、投资等账户信息）
- 个人账户信息（账户号码、密码、交易记录等）
- 个人信用信息（信用报告、信用评分等）
- 个人金融交易信息（转账、支付、投资等交易记录）
- 个人衍生信息（基于上述信息分析得出的消费习惯、投资偏好等）

### 二、C1-C5信息分类

根据《个人金融信息保护技术规范》，个人金融信息按敏感程度分为五个类别：

| 类别 | 名称 | 定义 | 示例 | 保护要求 |
|------|------|------|------|----------|
| C1 | 用户鉴别信息 | 用于验证用户身份的密钥、密码等 | 银行卡密码、网银登录密码、交易密码 | 最高级别保护 |
| C2 | 可识别特定个人的金融信息 | 可直接识别个人身份的信息 | 身份证号、银行卡号、手机号 | 高级别保护 |
| C3 | 机构内部信息资产 | 与个人金融活动相关的机构内部数据 | 客户风险评级、客户画像标签 | 中级别保护 |
| C4 | 一般性个人金融信息 | 公开或半公开的个人金融信息 | 公示的账户信息、产品持有情况 | 一般保护 |
| C5 | 公开信息 | 已公开的个人金融信息 | 已公开的上市公司股东信息 | 基本保护 |

### 三、信息收集规范

- **合法正当必要原则**：收集个人金融信息应当有明确、合理的目的，限于实现处理目的的最小范围。
- **明示同意原则**：收集个人金融信息前应当告知消费者并征得其明确同意。
- **最小化原则**：不得过度收集与业务无关的个人信息。

### 四、信息存储规范

- 个人金融信息应当存储在中华人民共和国境内（特殊情形除外）。
- 应当采取加密、脱敏等技术措施保障信息安全。
- 信息保存期限应当符合法律法规和业务需要。

### 五、信息使用规范

- 个人金融信息的使用应当限于收集时声明的目的范围。
- 超出原始目的使用的，应当重新取得消费者的同意。
- 不得将个人金融信息用于未经授权的营销推广。

### 六、信息共享规范

- 向第三方共享个人金融信息前，应当取得消费者的单独同意。
- 应当对第三方进行信息安全评估。
- 应当与第三方签订信息安全保护协议。

## 零售业务相关条款解读

### （一）开户环节的信息保护

**1. 信息收集的合规要求**

开户环节是个人金融信息收集的第一个节点，需要：
- 在开户申请表中明确告知信息收集的目的、范围和方式
- 以显著方式获取客户的明确同意（如勾选同意框、签署同意声明）
- 仅收集开户所必需的信息，不得过度收集
- 告知客户信息的保存期限和使用范围

**2. 信息存储的安全要求**

- 客户身份证件复印件应加密存储
- 客户签名影像应进行安全保护
- 开户信息应进行脱敏处理后用于分析
- 纸质资料应在专用保管柜中存放，设专人管理

### （二）贷款业务的信息保护

**1. 贷前调查**

- 征信信息查询须取得借款人的书面授权
- 查询记录应当完整保存
- 不得将征信信息用于贷款审批以外的目的
- 借款人提供的收入证明、财产证明等资料应妥善保管

**2. 贷中管理**

- 贷款审批信息仅限相关人员知悉
- 借款人的个人信息不得在审批流程中不当扩散
- 贷款合同中的个人信息应加密存储

**3. 贷后管理**

- 贷后检查中获取的客户信息应保密
- 催收过程中不得向无关第三人透露借款人的债务信息
- 不良贷款的转让和核销涉及客户信息时，应做好脱敏处理

### （三）理财销售的信息保护

**1. 风险评估信息**

- 客户风险承受能力评估的结果属于敏感信息
- 评估问卷及结果应加密存储
- 不得将客户风险评估信息用于营销分析以外的目的

**2. 持仓信息**

- 客户的理财产品持仓信息属于C2级别信息
- 不得向未经授权的第三方披露客户的持仓信息
- 在客户服务中查询持仓信息时须验证客户身份

**3. 交易信息**

- 客户的理财交易记录属于C2级别信息
- 交易信息应进行加密存储和传输
- 不得将交易信息用于客户授权以外的分析

### （四）银行卡业务的信息保护

**1. 卡号信息**

- 银行卡号属于C2级别信息
- 在各类凭证和界面中应进行部分脱敏（如仅显示后四位）
- 银行卡号不得在公共场所明文展示

**2. 密码信息**

- 银行卡密码属于C1级别信息（最高保护级别）
- 密码在传输过程中必须加密
- 密码不得以明文形式存储在任何系统中
- 银行工作人员不得询问或记录客户密码

**3. 交易记录**

- 交易记录属于C2级别信息
- 短信/推送通知中应进行部分脱敏
- 交易记录的查询须验证客户身份

### （五）电子渠道的信息保护

**1. 手机银行/网上银行**

- 登录密码、交易密码等C1级别信息须加密传输和存储
- 客户端应防止信息被截屏或复制
- 会话管理应设置超时自动退出
- 敏感操作应进行二次验证

**2. 短信通知**

- 短信中的账户信息应进行脱敏处理
- 不得在短信中展示完整的银行卡号或账户余额
- 短信内容不得泄露客户的交易对手信息

**3. 客户服务中心**

- 客服人员查询客户信息前须验证客户身份
- 通话录音中涉及的敏感信息应进行脱敏处理
- 不得在公共场合讨论客户的个人信息

### （六）信息安全事件管理

**1. 事件分类**

| 级别 | 描述 | 响应要求 |
|------|------|----------|
| 特别重大 | 泄露C1/C2信息超过100万条 | 2小时内报告监管机构 |
| 重大 | 泄露C1/C2信息10-100万条 | 24小时内报告监管机构 |
| 较大 | 泄露C1/C2信息1-10万条 | 48小时内报告监管机构 |
| 一般 | 泄露C1/C2信息不足1万条 | 72小时内报告监管机构 |

**2. 应急响应**

- 立即启动应急预案
- 采取技术措施阻止信息继续泄露
- 评估泄露范围和影响
- 向监管机构报告
- 通知受影响的消费者
- 配合调查和整改

## 对城商行/农商行的影响

### 1. 信息安全建设投入加大

城商行、农商行在信息安全方面的投入相对不足：
- 需要升级信息安全基础设施
- 需要部署数据分类分级管理系统
- 需要建立信息安全监控和预警体系
- 需要配备专业的信息安全人才

### 2. 业务流程改造

个人金融信息保护要求对现有业务流程进行改造：
- 在各业务环节增加信息收集和使用的授权环节
- 对系统中的客户信息进行分类和脱敏处理
- 调整客户服务流程中的身份验证环节
- 完善信息共享的审批和记录机制

### 3. 第三方管理

城商行、农商行与第三方合作时需要：
- 对合作方进行信息安全能力评估
- 在合作协议中明确信息安全保护责任
- 对合作方的信息使用情况进行持续监控
- 建立合作方信息安全事件的联合应急机制

### 4. 合规成本增加

- 需要投入人力和财力进行信息分类分级工作
- 需要开展全员信息安全培训
- 需要定期进行信息安全审计和评估
- 需要建立信息安全事件的应急和报告机制

## 合规实施要点

### 1. 制度建设

| 制度名称 | 关键内容 |
|----------|----------|
| 个人金融信息保护管理办法 | 总体框架、职责分工、管理原则 |
| 个人金融信息分类分级管理办法 | C1-C5分类标准、分级保护要求 |
| 个人金融信息收集管理办法 | 收集原则、授权管理、最小化原则 |
| 个人金融信息共享管理办法 | 共享审批、第三方评估、协议管理 |
| 个人信息安全事件管理办法 | 事件分类、应急响应、报告流程 |
| 个人金融信息安全审计制度 | 审计范围、审计频率、整改跟踪 |

### 2. 技术措施

- **数据加密**：对C1、C2级别信息进行存储和传输加密
- **数据脱敏**：在非生产环境和业务展示中使用脱敏数据
- **访问控制**：建立基于角色的数据访问权限管理
- **安全审计**：对敏感信息的访问行为进行完整记录和审计
- **安全监控**：部署异常行为检测和告警系统

### 3. 人员管理

- 对接触敏感信息的人员进行背景调查
- 签订保密协议
- 定期开展信息安全培训
- 建立信息安全违规的问责机制
- 离岗人员的信息权限及时回收

### 4. 第三方管理

- 建立第三方信息安全评估标准
- 与第三方签订信息安全保护协议
- 定期评估第三方的信息安全状况
- 建立第三方信息安全事件的通知和处置机制

## 监管处罚案例

### 案例一：某城商行客户信息泄露案（2023年）

- **违规事实**：该城商行的客户管理系统存在安全漏洞，导致约30万条个人客户的姓名、手机号、银行卡号等信息被黑客窃取并在暗网出售。
- **处罚结果**：被人民银行罚款500万元，信息科技部门负责人被免职。
- **涉及条款**：个人信息安全保护相关要求。

### 案例二：某农商行违规查询征信信息案（2022年）

- **违规事实**：该农商行信贷人员利用工作便利，在未获得客户授权的情况下查询客户个人征信报告，并将部分征信信息提供给第三方营销机构。
- **处罚结果**：被罚款100万元，涉事信贷人员被解除劳动合同。
- **涉及条款**：征信信息查询须取得书面授权。

### 案例三：某银行过度收集个人信息案（2023年）

- **违规事实**：某银行手机银行APP在安装时索取了通讯录、通话记录、短信内容等与银行业务无关的权限，过度收集了客户个人信息。
- **处罚结果**：被工信部通报并要求整改，同时被人民银行罚款80万元。
- **涉及条款**：最小化原则。

### 案例四：某银行员工倒卖客户信息案（2021年）

- **违规事实**：某银行客户经理利用职务便利，将约5万条高净值客户的个人信息（包括资产信息、联系方式等）出售给第三方理财公司，获利约30万元。
- **处罚结果**：涉事员工被以侵犯公民个人信息罪判处有期徒刑3年，银行被罚款200万元。
- **涉及条款**：个人金融信息保护、从业人员行为规范。
