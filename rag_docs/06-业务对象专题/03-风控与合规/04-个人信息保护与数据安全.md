# 个人信息保护与数据安全

## 专题概述

个人信息保护与数据安全是零售银行合规经营的底线工程，也是赢得客户信任的基础保障。银行作为金融数据的密集采集者和处理者，掌握着客户最敏感的身份信息、财务信息、交易信息和行为信息，一旦发生数据泄露或滥用，不仅会给客户造成严重损失，也将使银行面临严厉的监管处罚和声誉损害。

近年来，我国数据安全法律体系加速完善。《个人信息保护法》（2021年11月实施）、《数据安全法》（2021年9月实施）、《银行保险机构数据安全管理办法》（2024年12月发布）等法律法规相继出台，构建了从个人信息保护到数据安全管理的完整法律框架。2025年11月，中国人民银行进一步发布《中国人民银行业务领域数据安全管理办法》，对金融数据安全提出更细化的要求。

对城商行/农商行而言，数据安全合规面临特殊挑战：一是IT基础设施相对薄弱，数据安全防护能力有限；二是数据治理体系不够完善，数据分类分级尚未全面落实；三是员工数据安全意识参差不齐，内部泄露风险不容忽视；三是第三方合作方管理难度大。建立系统化的个人信息保护与数据安全管理体系，是中小银行数字化转型的必修课。

### 二、数据分类体系

根据《银行保险机构数据安全管理办法》，银行数据分为以下类别：

| 数据类型 | 说明 | 典型数据 |
|----------|------|----------|
| 客户数据 | 客户身份、账户、交易等信息 | 姓名、证件号、账户余额、交易记录 |
| 业务数据 | 银行业务经营数据 | 业务量、收入、产品数据 |
| 管理数据 | 银行内部管理数据 | 组织架构、人事、财务 |
| 监管数据 | 向监管报送的数据 | 报表、统计、合规数据 |
| 系统数据 | IT系统运行数据 | 日志、配置、网络数据 |

### 三、个人信息敏感程度分级

根据监管要求，个人信息按敏感程度分为不同级别（部分银行采用C1-C5分级体系）：

| 级别 | 名称 | 定义 | 典型数据 | 保护要求 |
|------|------|------|----------|----------|
| C1 | 公开级 | 可公开的信息 | 银行公告、网点信息 | 基本保护 |
| C2 | 内部级 | 仅限内部使用 | 内部通知、培训材料 | 内部控制 |
| C3 | 敏感级 | 泄露会造成一定影响 | 员工信息、一般业务数据 | 加强保护 |
| C4 | 高敏级 | 泄露会造成严重影响 | 客户姓名+证件号、账户信息 | 严格保护 |
| C5 | 核心级 | 泄露会造成极其严重影响 | 交易密码、生物特征、大额交易 | 最高保护 |

## 详解内容

### 一、个人信息收集与使用规范

#### 收集的合法基础

银行收集个人信息须基于以下合法基础之一：

1. **取得个人同意**：最常见的合法基础
2. **为订立/履行合同所必需**：如开户、贷款
3. **为履行法定义务所必需**：如反洗钱、税务报告
4. **为应对突发公共卫生事件所必需**
5. **为公共利益实施新闻报道、舆论监督**

#### 收集原则

| 原则 | 说明 | 实践要求 |
|------|------|----------|
| 合法正当必要 | 仅收集实现目的所必需的最少信息 | 不收集与服务无关的信息 |
| 知情同意 | 充分告知后获得客户同意 | 隐私政策明示收集目的和范围 |
| 目的限定 | 收集的个人信息仅用于声明的目的 | 不得超范围使用 |
| 公开透明 | 处理规则公开，操作可查 | 提供信息查询和导出功能 |

#### 敏感个人信息的特殊要求

以下信息属于"敏感个人信息"，须获得**单独同意**：

| 敏感信息类型 | 银行场景 | 获取方式 |
|-------------|----------|----------|
| 生物特征信息 | 人脸识别、指纹 | 单独弹窗同意 |
| 金融账户信息 | 账户余额、交易记录 | 单独弹窗同意 |
| 行踪轨迹 | 位置信息 | 单独弹窗同意 |
| 14岁以下未成年人信息 | 亲子账户 | 监护人单独同意 |
| 征信信息 | 征信查询 | 单独签署授权书 |

### 二、数据分类分级保护制度

#### 数据分类分级流程

```
数据资产梳理 → 数据分类 → 数据分级 → 确定保护策略 → 实施保护措施 → 持续监控
```

#### 各级数据保护措施

| 级别 | 存储保护 | 传输保护 | 使用保护 | 销毁要求 |
|------|----------|----------|----------|----------|
| C1公开级 | 基本访问控制 | 无特殊要求 | 无特殊要求 | 常规删除 |
| C2内部级 | 内部访问权限 | 内部网络传输 | 使用记录 | 安全删除 |
| C3敏感级 | 加密存储 | 加密传输 | 审批+记录 | 安全销毁 |
| C4高敏级 | 高强度加密+分区 | 端到端加密 | 最小权限+审批 | 彻底销毁 |
| C5核心级 | 硬件加密+隔离 | 专线+加密 | 双人审批+全程审计 | 物理销毁 |

### 三、数据脱敏技术

#### 脱敏场景

| 场景 | 脱敏要求 | 脱敏方式 |
|------|----------|----------|
| 柜面显示 | 证件号/手机号部分隐藏 | 静态脱敏（如：138****5678） |
| 开发测试 | 使用脱敏数据 | 静态脱敏（替换/扰乱） |
| 数据分析 | 统计用途不需明细 | 聚合/泛化 |
| 第三方共享 | 最小必要原则 | 静态脱敏+去标识化 |
| 日志记录 | 不得记录敏感信息 | 实时脱敏 |
| 客服展示 | 必要信息脱敏展示 | 动态脱敏 |

#### 脱敏规则

| 数据项 | 脱敏规则 | 示例 |
|--------|----------|------|
| 姓名 | 保留姓，名用*代替 | 张** |
| 身份证号 | 保留前3后4 | 110***********1234 |
| 手机号 | 保留前3后4 | 138****5678 |
| 银行卡号 | 保留前4后4 | 6225****1234 |
| 地址 | 保留省市，详细用* | 北京市**区**** |
| 交易金额 | 区间化/模糊化 | 10-50万元 |

### 四、数据安全全生命周期管理

#### 数据采集

- 仅采集业务必需的数据
- 获取客户知情同意
- 验证数据来源合法性
- 记录采集日志

#### 数据存储

- 按安全级别分区存储
- 敏感数据加密存储（AES-256或以上）
- 建立备份和容灾机制
- 定期清理过期数据

#### 数据使用

- 最小权限原则（Need to Know）
- 敏感数据访问须审批
- 数据使用行为审计
- 防止数据批量导出

#### 数据传输

- 敏感数据传输加密（TLS 1.2+）
- 跨网络传输安全审查
- 第三方数据传输安全评估
- 跨境传输合规审批

#### 数据销毁

- 过期数据及时销毁
- 存储介质安全销毁
- 销毁过程记录留痕
- 备份数据同步销毁

### 五、跨境数据传输

#### 跨境传输的合规要求

根据《个人信息保护法》和《数据安全法》：

| 合规路径 | 适用条件 | 说明 |
|----------|----------|------|
| 安全评估 | 关键信息基础设施运营者/处理大量个人信息 | 经网信部门安全评估 |
| 标准合同 | 中小规模个人信息出境 | 与境外接收方签订标准合同 |
| 认证 | 特定行业 | 经专业机构认证 |

#### 银行跨境传输场景

| 场景 | 是否涉及跨境 | 合规要求 |
|------|-------------|----------|
| SWIFT跨境汇款 | 是（交易信息出境） | 安全评估/标准合同 |
| 境外分支机构数据共享 | 是 | 安全评估 |
| 使用境外云服务 | 是（数据可能出境） | 使用境内云或安全评估 |
| 接入境外第三方服务 | 视情况 | 评估数据存储位置 |
| 客户自行在境外使用手机银行 | 否（客户自主行为） | 不涉及 |

### 六、数据泄露应急响应

#### 应急响应流程

```
发现泄露 → 初步评估 → 启动应急 → 控制扩散 → 调查原因 → 影响评估 → 通知相关方 → 整改复盘
（0小时）  （2小时内）  （4小时内）  （24小时内）  （72小时内）  （持续）    （72小时内）  （30天内）
```

#### 各环节详细要求

**发现与报告（0-2小时）**：
- 任何人发现疑似数据泄露应立即报告信息安全部门
- 初步判断泄露类型、范围、严重程度
- 向分管领导报告

**启动应急（2-4小时）**：
- 成立应急小组（信息安全+业务+法务+公关）
- 评估泄露事件等级
- 制定应急处置方案

**控制扩散（4-24小时）**：
- 封堵泄露源头（关闭漏洞、更改密码、隔离系统）
- 防止数据进一步扩散
- 保存证据

**通知相关方（72小时内）**：
- 通知监管部门（人民银行/金融监管总局）
- 通知受影响客户（告知泄露范围、风险、建议措施）
- 必要时通知公安机关

**整改复盘（30天内）**：
- 查明泄露原因
- 修补安全漏洞
- 完善管理制度
- 开展员工培训

#### 泄露事件分级

| 级别 | 标准 | 响应层级 |
|------|------|----------|
| 特别重大 | 100万人以上个人信息泄露 | 总行+监管报告 |
| 重大 | 10万-100万人信息泄露 | 总行级别响应 |
| 较大 | 1万-10万人信息泄露 | 分行级别响应 |
| 一般 | 1万人以下信息泄露 | 部门级别响应 |

## 管理方法与工具

### 1. 数据安全管理平台

| 功能模块 | 说明 |
|----------|------|
| 数据资产管理 | 数据资产清单、分类分级标注 |
| 权限管理 | 细粒度数据访问权限控制 |
| 脱敏管理 | 脱敏规则配置、脱敏效果验证 |
| 审计管理 | 数据访问行为审计、异常行为检测 |
| 合规管理 | 隐私政策管理、同意记录管理 |
| 应急管理 | 应急预案管理、事件处置跟踪 |

### 2. 隐私政策管理

隐私政策（个人信息处理规则）应包含：

- 信息处理者的名称和联系方式
- 个人信息的处理目的、处理方式
- 处理的个人信息种类
- 信息的保存期限
- 个人行使权利的方式和程序
- 向第三方提供信息的规则
- 跨境提供信息的规则

### 3. 第三方数据安全管理

| 管理环节 | 要求 |
|----------|------|
| 准入评估 | 对第三方数据安全能力进行评估 |
| 合同约定 | 明确数据安全责任、使用范围、保密义务 |
| 数据传输 | 加密传输，最小必要原则 |
| 持续监控 | 定期审计第三方数据使用情况 |
| 退出管理 | 合同终止后要求第三方删除数据并确认 |

### 4. 员工数据安全管理

| 管理措施 | 说明 |
|----------|------|
| 安全培训 | 全员数据安全培训，每年不少于1次 |
| 权限管控 | 最小权限原则，离岗/离职及时回收 |
| 行为审计 | 记录员工数据访问和操作行为 |
| 保密协议 | 全员签署保密协议和竞业限制 |
| 违规处罚 | 明确数据安全违规的处罚措施 |

## 城商行/农商行实践

### 案例一：某城商行数据分类分级实施

该城商行按照监管要求全面推进数据分类分级：

**第一阶段（1-3月）——资产梳理**：
- 梳理全行数据资产，识别出2000+数据字段
- 按照监管标准完成数据分类（客户数据/业务数据/管理数据等）
- 按照敏感程度完成数据分级（C1-C5）

**第二阶段（4-6月）——保护策略制定**：
- 针对各级数据制定差异化保护策略
- 部署数据脱敏系统，对C4/C5级数据实施强制脱敏
- 升级数据访问权限控制，实施最小权限原则

**第三阶段（7-12月）——系统实施与优化**：
- 部署数据安全管理平台
- 建设数据访问行为审计系统
- 开展全员数据安全培训
- 建立数据安全定期检查机制

### 案例二：某农商行个人信息保护合规整改

该农商行在监管检查中被发现多项个人信息保护不合规问题，进行全面整改：

**问题清单**：
- 隐私政策过于笼统，未明确具体收集范围
- 手机银行未获取客户单独同意即收集位置信息
- 客服系统展示客户身份证号未脱敏
- 第三方合作机构数据安全评估缺失
- 员工可批量导出客户信息，缺乏审计

**整改措施**：
- 重新编写隐私政策，逐项明确收集目的和范围
- 手机银行增加敏感信息单独同意弹窗
- 客服系统实施动态脱敏
- 建立第三方数据安全评估机制
- 实施数据导出审批和审计制度

### 案例三：某城商行数据泄露应急演练

该城商行定期开展数据泄露应急演练：

- 模拟场景：内部员工泄露客户交易数据
- 演练内容：事件发现→报告→应急处置→通知客户→整改复盘
- 参演部门：信息安全、零售业务、客服、法务、公关
- 演练频次：每半年一次
- 演练后评估：完善应急预案，提升响应速度

## 监管要求

### 1. 核心监管文件

| 文件名称 | 核心内容 |
|----------|----------|
| 《个人信息保护法》 | 个人信息处理的全面规范，同意机制、跨境传输 |
| 《数据安全法》 | 数据分类分级、安全评估、应急管理 |
| 《银行保险机构数据安全管理办法》（2024年） | 金融机构数据分类分级保护制度 |
| 《中国人民银行业务领域数据安全管理办法》（2025年） | 央行领域数据分类分级细化 |
| 《个人金融信息保护技术规范》（JR/T 0171-2020） | 技术层面的保护要求 |
| 《金融消费者权益保护实施办法》 | 消费者信息安全权保护 |

### 2. 银行法定义务

| 义务 | 法律依据 | 具体要求 |
|------|----------|----------|
| 告知义务 | 个人信息保护法 | 处理个人信息前须充分告知 |
| 同意义务 | 个人信息保护法 | 获得客户同意（敏感信息须单独同意） |
| 最小必要 | 个人信息保护法 | 仅收集和处理必需的信息 |
| 安全保障 | 数据安全法 | 采取技术和管理措施保障数据安全 |
| 泄露通知 | 个人信息保护法 | 发生泄露时通知监管和个人 |
| 影响评估 | 个人信息保护法 | 处理敏感信息前进行影响评估 |
| 分类分级 | 数据安全管理办法 | 建立数据分类分级保护制度 |

### 3. 违规处罚

| 违规情形 | 处罚措施 |
|----------|----------|
| 未经同意收集个人信息 | 警告→限期改正→罚款（最高5000万元或上年营收5%） |
| 数据泄露未及时通知 | 警告→罚款→暂停业务 |
| 超范围使用个人信息 | 警告→罚款→暂停业务 |
| 非法买卖个人信息 | 罚款→暂停业务→刑事责任 |
| 未履行数据安全保护义务 | 警告→罚款→暂停业务→高管问责 |

### 4. 监管检查重点

监管在数据安全方面的现场检查和非现场检查重点：

- 数据分类分级是否落实
- 个人信息收集是否合规（隐私政策、同意机制）
- 数据访问权限是否合理
- 数据脱敏是否到位
- 第三方数据共享是否合规
- 数据泄露应急预案是否建立并演练
- 员工数据安全培训是否开展
- 数据安全事件是否及时报告

## 常见问题

**Q1：银行的隐私政策客户根本不看，点了同意就算合规吗？**
A：从法律角度，客户点击"同意"按钮构成有效同意，但前提是隐私政策的内容清晰、易懂、不含误导。监管检查时会关注隐私政策是否以显著方式呈现、是否使用通俗语言、是否将重要条款突出显示。银行不能以"客户自己没看"为由推卸责任，应确保隐私政策设计合理且易于理解。

**Q2：银行可以查看客户的交易记录吗？需要什么条件？**
A：银行因业务需要可以查看客户交易记录（如风控监测、客户服务、投诉处理等），但须遵循"最小必要"原则，仅查看与业务目的相关的信息。敏感操作（如批量导出、跨部门查看）应有审批和审计机制。银行不得将客户交易信息用于未经授权的目的。

**Q3：第三方合作机构（如催收公司、技术供应商）可以使用客户信息吗？**
A：可以在合规前提下使用。银行作为信息处理者，向第三方提供客户信息须：一是获得客户同意（除非法律另有规定）；二是与第三方签订数据安全协议；三是仅提供业务必需的最少信息；四是对第三方进行数据安全能力评估；五是监督第三方数据使用情况。

**Q4：城商行/农商行数据安全合规的最大挑战是什么？**
A：主要挑战包括：一是数据分类分级工作量大、技术能力不足；二是员工数据安全意识薄弱，内部泄露风险高；三是第三方合作管理难度大；四是安全投入有限，防护能力与威胁不匹配；五是专业人才匮乏。建议采取"监管驱动、分步推进、重点突破"的策略，优先解决最紧迫的合规问题。

**Q5：客户要求删除其个人信息，银行可以拒绝吗？**
A：部分可以拒绝。根据《个人信息保护法》，客户有权要求删除其个人信息，但以下情况银行可以拒绝：一是法律法规规定须保留的（如反洗钱要求保留5年、会计档案保留要求）；二是为实现合同目的所必需的（如贷款未还清前的账户信息）；三是其他法定保留义务。银行应向客户说明拒绝理由和保留期限。
